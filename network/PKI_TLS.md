# PKI(Public Key Infrastructure)
## 構成要素
- RA(Registration Authority)/登録局
  - 証明書申請者の身元審査
- CA(Certficate Authority)/認証局
  - 証明書を発行する信頼機関
  - 役割
    - サイト運営者が本物か確認
    - 確認後に証明書を発行
    - 自分の署名をつける
  - ブラウザは、証明書に信頼できるCAが署名しているかどうかを確認
- リポジトリ
  - 発行された証明書や、無効になった証明書リストの公開場所


## 証明書
### 概要
- 公開鍵と、公開鍵を渡してきた相手の身元を結びつけるもの
- 対象のドメイン名に対して、手元の公開鍵が対応している、この事を信頼された第三者が署名して保証したもの
- X.509: 国際規格

### 内容
#### 基本領域
- バージョン
- シリアル番号
- 署名アルゴリズム
- 発行者(Issure)
- 主体者(Subject)
- 有効期間
- 公開鍵
- 署名値
#### 拡張領域
- SAN(Subject Aleternative Name)
  - Subject欄のCN(Common Name)には一つしかドメインをかけない
  - 代わりにここに有効なドメイン名をリストアップ
  - クライアントは、証明書の公開鍵を信頼する場合、SAN内の証明書も公開鍵として信頼する
- 鍵使用法(Key Usage)
  - 公開鍵の使用制限
...etc

## 署名済み証明書の作成と検証
### 作成(認証局が行う)
1. 証明書の本文αを、予めCA経由で共通されているデータγを元にハッシュ化(指紋S)
2. 1を秘密鍵で暗号化(署名β)
3. 本文αと署名βのセットをサーバに付与(これが署名済み証明書)

### 検証
1. クライアントは署名済み証明書の署名βを公開鍵で開き、指紋Sを生成
2. 本文αと共通データγでハッシュ化した指紋Cを作成
3. 指紋Sと指紋Cを照合

### サーバがクライアントに送る内容
- サイト名
- サイトの公開鍵
- 有効期限
- 署名済み証明書


## 信頼の連鎖(Chain of Trust)
### 前提知識
- クライアントは、証明書の検証をするとき、クライアントは単一の証明書だけではなく、証明書チェーンをたどる(a?b?)
- ルートCA
  - 署名の起点となる、中間CAに署名をした機関
  - 普段はオフラインで、物理的にセキュリティで守られている
- 中間CA
  - ルートCAから信頼されたCA
  - 日々の発行業務を行う
- Root Store
  - ブラウザやOSが最初から信頼している、Root CAのリスト

### パス検証（Path Validation）
以下の連鎖をたどって、信頼しているルートCAに繋がるか確認
1. ブラウザやOSはルートCAを信頼している
2. ルートCAが中間CAを信頼して署名
3. 中間CAがサーバ証明書を発行して署名
4. サーバ証明書にはサイトの公開鍵が入っている

### OSやブラウザが「信頼」を判定するプロセス
- 信頼アンカーへの到達
  - クライアントは、自身のOSやアプリが持っている「信頼済みルート証明書ストア(Root Store)」を検索
    - 受け取った証明書αに署名した中間CA(証明書αのIssuer)が、Root Storeに乗っているか調べる
- 証明書の有効性チェック
  - 有効期限等
- ホスト名の検証
  - SANに、アクセス対象のドメインが含まれるかチェック


# TLS(Transport Layer Security) ※1.3
## 概要
- 以下の提供
    - 暗号化
    - 完全性
    - 認証
- HTTPS
  - 先にTLSで安全性を確保してからHTTP通信を行う
- 以下の順番で、保護された通信経路を作る
1. ハンドシェイク
2. レコードプロトコル


## ハンドシェイク手順
### 流れ
1. 準備
  - ClientHello
  - ServerHello
2. 認証
  - EncryptedExtensions
  - Certificate
  - CertificateVerify
3. 終了確認
  - Finished

### 準備
#### ClientHello
- サポートする TLS バージョン
- サポートする Cipher Suites
- ランダム値（Client Random）
- 拡張（extensions）
- SNI（Server Name Indication）
- 鍵共有用の KeyShare 拡張
- サポートする署名アルゴリズム
- ALPN などの追加拡張
#### ServerHello
- 選択した TLS バージョン
- 選択した Cipher Suite
- Server Random
- 選択した鍵共有グループ
- サーバ側の KeyShare

### 認証
#### EncryptedExtensions
- ALPN の選択結果
- その他サーバ側拡張応答
#### Certificate
- PKIによるチェック
#### CertificateVerify
- 直前までのハンドシェイク文脈に対して署名を行う
- 署名は証明書に対応する秘密鍵から生成
- クライアントは、証明書に含まれる公開鍵に対応する秘密鍵をサーバが現在所持している事を確認

### 終了確認
#### Finished
- ここまでのハンドシェイクメッセージ全体に対して、サーバは導出済み鍵を用いた検証を行う
- クライアントは改ざんや双方の鍵の所持を確認



# SSL(Secure Sockets Layer)インスペクション
## 概要/仕組み
- プロキシサーバーが、暗号化されたTLS経由の中身を検閲するための技術
- プロキシはクライアントとサーバの間に立ち、2つの独立したTLSコネクションを作る
- サーバとの通信ではクライアント、クライアントとの通信ではサーバとして振る舞う
- 中間攻撃者と同じ手法でサーバから送られてきた情報の中身を検閲してから、クライアントに送る
  - この際本来プロキシはサーバの秘秘密を持ちえないため、TLSのCertificateVerifyの突破が不可能
- そのため以下の手順で、この検閲を実現
  - SeverHelloの段階で、プロキシはクライアントに対して、自身がサーバのように振る舞ってTLS通信を行う
  - この際の証明書はもちろん偽物だが、PCのRoot Storeに予めプロキシを登録しておくことで、この偽の証明書を信頼させる

## 偽装工程
- 通信
  - プロキシは、サーバから届いた証明書をコピーして、その情報を元にサーバとしてクライアントと通信する
- 証明書
  - コピーした情報を使って、偽物のその場限りの証明書を生成
    - インターセプション証明書、または動的生成証明書という
  
  
# OSによる証明書管理の仕組みの違い(win)
## Windows vs Linux(WSL2) 証明書管理の比較
| 項目 | Windows (Host) | Linux / WSL2 (Ubuntu等) |
| :--- | :--- | :--- |
| **管理方式** | **中央集権データベース** | **分散ファイルシステム** |
| **保存形式** | レジストリ / バイナリ形式 | PEM形式 (`.crt`, `.pem`) / テキスト |
| **主な管理パス** | `certmgr.msc` (GUIツール) | `/etc/ssl/certs/` (ディレクトリ) |
| **集約ファイル** | (OSが動的に処理) | `/etc/ssl/certs/ca-certificates.crt` |
| **更新コマンド** | GPO / Windows Update | `sudo update-ca-certificates` |
| **主な参照ツール** | Chrome, Edge, Outlook, etc. | curl, wget, rbenv, apt, Git, etc. |
| **独立性** | WSL側の変更に影響されない | Windows側の追加を自動検知しない |
